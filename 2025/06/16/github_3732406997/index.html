enum_chrome module SYSTEM权限下崩溃漏洞

漏洞信息

漏洞名称： enum_chrome module SYSTEM权限下崩溃漏洞

漏洞类型： 权限绕过

漏洞等级： 中危

漏洞描述： 在metasploit-framework的enum_chrome模块中，当以SYSTEM权限运行时会导致崩溃，并且会话处于不可用状态，直到用户执行rev2self命令恢复steal_token调用。此问题在64位的Server 2019实例上被发现，且在使用#18558的更改后出现。通过禁用token窃取和迁移逻辑可以临时解决此问题，但可能影响自动解密功能。

产品厂商： rapid7

产品名称： metasploit-framework

来源： https://github.com/rapid7/metasploit-framework/issues/18559

类型： rapid7/metasploit-framework:github issues

来源概述

When running the post/windows/gather/enum_chrome module, it will crash when run as NT AUTHORITY\SYSTEM. I noticed this issue while making the changes in #18558. After the module fails, it should also be noted that the sessions is left in a pretty unusable state until the user runs rev2self to revert the steal_token call.

The target is a 64-bit Server 2019 instance and the changes from #18558 are loaded on my test branch.

After implementing the following quick workaround the module runs as NT AUTHORITY\SYSTEM but with a warning about automatic decryption not being possible. The module also works as intended after migrating into a process running as a local admin.

Testing Output

I’m not sure why the token impersonation or automatic migration is required. But it looks like removing it causes the module work work as I would expect it to work. It should be noted that if the session is running as SYSTEM and the module migrates into a process running as a user, it won’t be able to migrate back into the SYSTEM process due to the loss of privileges. My guess is that the MIGRATE option was originally intended to be used to migrate into the explorer.exe process owned by the current non-SYSTEM user.

Further testing for edge cases should probably take place before just removing the token impersonation and migration logic.
